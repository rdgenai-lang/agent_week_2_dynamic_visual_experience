Lesson 05 · Evaluation

Evaluating your agents

An agent that works once isn't done — it has to keep working after every prompt tweak, model swap, and new tool. Let's see how to measure did it succeed, how efficiently, how fast, and how to wire all of it into your GitHub repo.

PART 01

Why agents need evaluation

Non-deterministic = "works on my machine" doesn't mean anything

Unlike regular code, an LLM can give a different answer to the same input. Tweak one word in the prompt and behaviour shifts in ways you can't predict. The only way to know if today's agent is better than yesterday's is to run a fixed set of tests on both and compare the scores.

▶ See why agents need measurement, not gut feel
Without eval: "it felt better" With eval: "v1.3 scores 87% vs 82% — ship it" Bonus: catches regressions before users do
PART 02

Task completion — did it actually work?

Golden test set + LLM-as-judge = pass rate

Build a golden dataset: a list of inputs paired with the right answer (or rubric). Run your agent on each input, then ask a stronger LLM to grade whether the agent's output matches. The fraction it gets right is your pass rate — your single most important number.

▶ Watch one test case go through the judge
from langsmith.evaluation import evaluate from langchain.evaluation import load_evaluator judge = load_evaluator("labeled_score_string", llm=stronger_llm) results = evaluate( my_agent, data="golden-dataset-v1", evaluators=[judge], ) print(results.aggregate_results) # → {pass_rate: 0.87, ...}

PART 03

Tool-call efficiency

Same answer in 2 tool calls beats the same answer in 9

Two agents can both get the right answer but one spends 9 tool calls wandering while the other does it in 2. Each call costs tokens, time, and rate limits. Track tool calls per task, redundant calls (same args twice), and wrong tool picks — efficiency is a first-class metric.

▶ Compare wasteful vs efficient on the same task
Track: total_tool_calls per task Track: duplicate_calls (same name + same args) Track: wrong_tool rate (tool didn't help) Fix: sharpen tool docstrings · cache results
PART 04

Latency — how long users wait

Don't average. Look at p50 and p95.

Average response time hides the worst cases. p50 (median) is what half your users see; p95 is what the unluckiest 5% suffer. Break latency down per node — usually one slow node (long LLM call, slow API) is dragging the whole graph down.

PART 05

Structuring it in GitHub

Repo layout · evals as CI · dataset versioning

Don't keep evals in a notebook. Put your agent, tools, datasets, and eval scripts inside one repo with a clean folder layout. Then wire GitHub Actions to run evals on every PR — the metrics get posted as a comment, so reviewers see before/after pass rate, latency, and tool-call count without leaving GitHub.

▶ See evals run automatically on a PR
# .github/workflows/eval.yml — runs on every PR name: agent-evals on: [pull_request] jobs: evaluate: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: pip install -r requirements.txt - run: python evals/run.py --dataset evals/datasets/golden_v1.jsonl - uses: actions/github-script@v7 # post metrics back to PR with: script: | github.rest.issues.createComment({ ...require('./evals/report.js') })
DO

Good repo hygiene

  • Version your dataset (golden_v1.jsonl, v2…)
  • Eval script is just python evals/run.py
  • Block merges if pass rate drops >3pp
  • Cache LLM responses in CI to save cost
  • Store agent traces — debug failures by replay
AVOID

Common traps

  • Evals only in someone's notebook
  • Editing the dataset to make tests pass
  • One metric only — track 3+ at minimum
  • No baseline — always compare to main
  • Forgetting cost — pass rate ≠ "good agent"
